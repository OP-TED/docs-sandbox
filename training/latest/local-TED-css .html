How to make your local dev docs website look and function just like the production site

The css and other site functionality comes from two sources:

  • The antora.eui repository: the full ccs, javascript, etc elements provided in a release zip file.

  • The OP-TED.github.io (supplemental-ui), small modifications or Asciidoc elements that for some reason can’t be placed in the antora.eui release file.

The site generation process

  1. fetches the latest release ui-bundle zip file from the OP-TED/antora-eui repository (the location is specified in the antora-playbook.yml file).

  2. Fetches the supplemental-ui settings from the site generation repo and applies these on top of the antora-eui ones, overwriting or adding to them.

This process does not work entirely when generating a site locally, despite the settings being in the local-antora-playbook.yml (not sure why) but this is what you need to do to reproduce them:

  1. If you have never run the local site generation process locally, do so now in a terminal window from the docs-sandbox. Do this in VSC from within the docs-sandbox repository.
    antora local-antora-playbook.yml
    This will generate a new folder, “build” in the file structure.

  2. Download the latest ui-bundle.zip file from the antora-eui repo. Unzip it on your machine and copy the contents to the folder under “build” that has just “_” as its name. You will see a set of similar files, just overwrite these.

  3. Now go to the OP-tED.github.io repository download the custom.css file from the supplemental-ui/css folder. Put this into the css folder : build/_/css in the docs-sandbox.

The local-antora-playbook is currently set to look for the ui-bundle.zip and custom.css in two files under the content folder but as mentioned, the generation does not work properly. But… a ui bundle needs to be specified, so the settings still need to be in the yml file.

Note: When you clone docs-sandbox, neither the build folder nor the local-antora-playbook.yml will be cloned because they should be added to the .gitignore file locally and not pushed to the remote repository.

Local-antora-playbook.yml

site:
    title: TED Sandbox Documentation
    url: https://docs.ted.europa.eu
    start_page: home::index.adoc

asciidoc:
    attributes:
        ### Site sub-title (displayed next to the site.title at the header)
        page-site-subtitle: Documentation for TED developers and eSenders
        ### Component's Latest Version
#
        # We want to know the latest version of each component so we can create
        a redirect for the latest version.

        # Such a redirect would be useful for users that want to bookmark a link to the latest documentation for a component.

        # For example, we would be able to provide a link of the form:
https://docs.ted.europa.eu/eforms/latest.

        # It is also nice to be able to indicate a component's version in the main navigation panel (modules/ROOT/nav.adoc).

#

        # However, we do not currently have any other way to determine the latest version of a component outside a page in the component itself.

        # So, as a workaround, we store the latest version of each component in these site-wide asciidoc attributes.

        # The downside is that we have to update these attributes whenever we release a new version of a component.
        #
        eforms_latest_version: '1.13'
        epo_latest_version: '5.0.0'
        espd_latest_version: '5.1.0'
        #

    # extensions:
        #- ./lib/apidoc/apidoc-asciidoc-extension

content:
    edit_url: false
    sources:
